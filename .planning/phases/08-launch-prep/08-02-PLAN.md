**3. Create `apps/web/app/api/gdpr/export/route.ts`:**

GET endpoint for instant GDPR data export. Follow the established API auth pattern:
```typescript
import { auth } from '@clerk/nextjs/server'
import { createServiceSupabaseClient } from '@/lib/supabase/service'

export async function GET() {
  const { userId } = await auth()
  if (!userId) {
    return Response.json({ error: 'Неоторизиран достъп' }, { status: 401 })
  }

  const supabase = createServiceSupabaseClient()

  // Fetch all user data in parallel
  const [chartsRes, readingsRes, horoscopesRes, userRes] = await Promise.all([
    supabase.from('charts').select('*').eq('user_id', userId),
    supabase.from('ai_readings').select('*').eq('user_id', userId),
    supabase.from('daily_horoscopes').select('*').eq('user_id', userId),
    supabase.from('users').select('*').eq('clerk_id', userId).single(),
  ])

  const exportData = {
    exportedAt: new Date().toISOString(),
    user: userRes.data ? {
      subscriptionTier: userRes.data.subscription_tier,
      createdAt: userRes.data.created_at,
    } : null,
    charts: chartsRes.data ?? [],
    aiReadings: readingsRes.data ?? [],
    dailyHoroscopes: horoscopesRes.data ?? [],
  }

  const json = JSON.stringify(exportData, null, 2)
  return new Response(json, {
    headers: {
      'Content-Type': 'application/json',
      'Content-Disposition': 'attachment; filename="celestia-data-export.json"',
    },
  })
}
```

Note: `ai_readings` table has a `user_id` column (verified from schema). `daily_horoscopes` references via `chart_id` — for horoscopes, query by joining with charts: get user's chart IDs first, then query daily_horoscopes by those chart_ids. Actually, check the schema: if daily_horoscopes has user_id, query directly. If it only has chart_id, fetch charts first and then use chart IDs. Read the daily-horoscopes.ts schema to determine the correct approach.

IMPORTANT: Read `packages/db/src/schema/daily-horoscopes.ts` and `packages/db/src/schema/ai-readings.ts` to verify column names and which columns to use for user-specific queries.

**4. Create `apps/web/app/api/gdpr/delete-account/route.ts`:**

POST for requesting deletion, DELETE for cancelling during grace period.

POST handler:
- Auth check (return 401 if not authenticated, Bulgarian error message)
- Calculate `now` and `scheduledDeletion = now + 30 days`
- Update users table: `deleted_at = now`, `deletion_scheduled_at = scheduledDeletion` where `clerk_id = userId`
- Return JSON with message and scheduled date

DELETE handler:
- Auth check
- Update users table: `deleted_at = null`, `deletion_scheduled_at = null` where `clerk_id = userId`
- Return JSON confirmation

Follow exact pattern from research code examples.

Vercel cron endpoint for hard-deleting accounts past grace period. Follow the daily-horoscope cron pattern:

**1. Create `apps/web/app/(protected)/settings/privacy/page.tsx`:**

Server component following the established pattern (same as settings/page.tsx):
```typescript
import { auth } from '@clerk/nextjs/server'
import { redirect } from 'next/navigation'
import { createServiceSupabaseClient } from '@/lib/supabase/service'
import { PrivacySettingsContent } from './PrivacySettingsContent'

export default async function PrivacySettingsPage() {
  const { userId } = await auth()
  if (!userId) redirect('/auth')

  const supabase = createServiceSupabaseClient()
  const { data: user } = await supabase
    .from('users')
    .select('deleted_at, deletion_scheduled_at')
    .eq('clerk_id', userId)
    .single()

  return (
    <PrivacySettingsContent
      deletedAt={user?.deleted_at ?? null}
      deletionScheduledAt={user?.deletion_scheduled_at ?? null}
    />
  )
}
```
