  <name>Task 1: Create Zod validation schemas for birth data</name>
  <files>
    apps/web/lib/validators/birth-data.ts
  </files>
  <action>
    Create comprehensive Zod schemas for birth data validation with Bulgarian error messages:

    1. **birthDataSchema**: Full schema for birth data
       - name: string, min 1, max 100, Bulgarian error "Molia, vavedete ime"
       - birthDate: string, regex YYYY-MM-DD format, refine to ensure date is in the past
       - birthTimeKnown: boolean
       - birthTime: string (HH:MM format), optional/nullable
       - approximateTimeRange: enum ['morning', 'afternoon', 'evening', 'night'], optional/nullable
       - cityId: uuid string, optional/nullable (null for manual entry)
       - cityName: string, min 1, "Molia, izberete naseleno miasto"
       - latitude: number, -90 to 90
       - longitude: number, -180 to 180
       - manualCoordinates: boolean, optional (true if user entered lat/lon manually)

    2. Add superRefine for conditional validation:
       - If birthTimeKnown is true, birthTime is required
       - If birthTimeKnown is false, approximateTimeRange is required
       - Error: "Molia, vavedete chas ili izberete priblizitelen period"

    3. **createBirthDataSchema**: Extends birthDataSchema (same fields for creation)

    4. **updateBirthDataSchema**: Partial of birthDataSchema for PATCH operations

    5. Export TypeScript type: `type BirthData = z.infer<typeof birthDataSchema>`

    All error messages in Bulgarian. Time ranges per CONTEXT.md:
    - morning: 06:00-12:00
    - afternoon: 12:00-18:00
    - evening: 18:00-24:00
    - night: 00:00-06:00

    Note: This schema is used in TWO contexts:
    - API routes: use `.safeParse()` for server-side validation
    - React Hook Form (Plan 04): use `zodResolver(birthDataSchema)` for client-side validation
    Both patterns are valid and use the same underlying schema.
  </action>
  <verify>
    - TypeScript compiles without errors
    - Schema rejects invalid dates (e.g., "not-a-date")
    - Schema rejects future dates
    - Schema requires time OR approximate range based on birthTimeKnown
    - Error messages are in Bulgarian
  </verify>
  <done>

<task type="auto">
  <name>Task 2: Create birth data CRUD API routes</name>
  <files>
    apps/web/app/api/birth-data/route.ts

  </files>
  <action>
    Create API routes for birth data operations:

    1. **/api/birth-data/route.ts**:

       **GET**: Retrieve all user's charts
       - auth.protect() first
       - Query Supabase `charts` table (RLS filters automatically)
       - Order by createdAt descending
       - Return JSON array

       **POST**: Create new chart
       - auth.protect() first
       - Parse JSON body, validate with createBirthDataSchema.safeParse()
       - If validation fails: 400 with `{ error: string, details: fieldErrors }`
       - Insert into charts table (user_id auto-set by RLS default)
       - Return 201 with created chart

    2. **/api/birth-data/[id]/route.ts**:

       **GET**: Retrieve single chart by ID
       - auth.protect()
       - Query by id (RLS ensures user owns it)
       - 404 if not found
       - Return chart object

       **PATCH**: Update chart
       - auth.protect()
       - Validate body with updateBirthDataSchema.safeParse()
       - Update where id matches (RLS ensures ownership)
       - Return updated chart

       **DELETE**: Delete chart
       - auth.protect()
       - Delete where id matches (RLS ensures ownership)
       - Return 204 No Content

    Error responses use Bulgarian messages:
    - Validation: "Nevalidni danni" with field details
    - Not found: "Dannite ne biaha namereni"
    - Server error: "Greshka pri zapazvane"

    Use Supabase client from lib/supabase/server.ts
  </action>
  <verify>
    - POST with valid data returns 201
    - POST with invalid data returns 400 with field errors
    - GET returns user's charts (empty array if none)
    - PATCH updates specified fields only
    - DELETE returns 204
    - All routes return 401/404 for unauthenticated requests
  </verify>
