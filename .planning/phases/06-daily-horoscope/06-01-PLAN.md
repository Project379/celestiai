    1. Create `packages/astrology/src/transit.ts` with two exported functions:

    **calculateDailyTransits(date: Date)**: Returns `TransitData` with `{ date: string, planets: Omit<PlanetPosition, 'house'>[] }`. Reuse the exact same pattern as `calculatePlanetPosition` in `calculator.ts` — same `sweph.calc_ut` call with `SEFLG_MOSEPH | SEFLG_SPEED`, same `PLANETS_ORDER` loop, same `getZodiacSign`/`longitudeToSignDegree`. Use noon UTC as the Julian Day time (standard for daily transits). The `date` field in the return is ISO date string 'YYYY-MM-DD'.

    **calculateTransitAspects(transits: TransitData, natalPlanets: PlanetPosition[])**: Returns `TransitAspect[]`. Cross-reference each transit planet against each natal planet. Use TIGHTER orbs than natal: define `TRANSIT_ASPECT_DEFINITIONS` locally with conjunction=3, sextile=2, square=2.5, trine=2.5, opposition=3 for inner planets (sun, moon, mercury, venus, mars), and conjunction=4, sextile=3, square=3.5, trine=3.5, opposition=4 for outer planets (jupiter, saturn, uranus, neptune, pluto). Determine applying/separating by comparing current orb to future orb (same logic as `isApplying` in aspects.ts but adapted for transit-vs-natal where one side has no speed — natal positions are fixed, so only the transit planet speed matters). Return `{ transitPlanet, natalPlanet, aspect, orb, applying }`.

    Export both types (`TransitData`, `TransitAspect`) and both functions.

    2. Update `packages/astrology/src/index.ts` to re-export from transit.ts:
    - Export types: `TransitData`, `TransitAspect`
    - Export functions: `calculateDailyTransits`, `calculateTransitAspects`

    3. Create `packages/db/src/schema/daily-transits.ts` following the pattern in `ai-readings.ts`:

    - Export the table, and inferred types `DailyTransit`, `NewDailyTransit`.

    4. Create `packages/db/src/schema/daily-horoscopes.ts`:
    - Table: `daily_horoscopes` with columns: `id` (uuid, defaultRandom, primaryKey), `chart_id` (uuid, notNull, references charts.id with onDelete cascade), `user_id` (text, notNull), `date` (text, notNull), `content` (text, notNull), `generated_at` (timestamp with timezone, defaultNow, notNull), `model_version` (text, notNull).
    - Add `uniqueIndex('daily_horoscopes_chart_date_idx').on(table.chartId, table.date)`.
    - Export the table and inferred types `DailyHoroscope`, `NewDailyHoroscope`.

    5. Update `packages/db/src/schema/index.ts` to re-export from both new schema files (dailyTransits, dailyHoroscopes, and their types).

    - Returns the system prompt string for Gemini. Use the Bulgarian mystical guide persona from the research (Celestia voice).
    - 4-6 paragraphs, mention 2-4 significant transit aspects, end with practical suggestion.
    - Include sentinel markers instruction: `[planet:KEY]Name[/planet]` using same keys as Oracle.
    - All output in Bulgarian Cyrillic.
    - Use the exact prompt from the research section as the baseline.

    2. Create `apps/web/lib/horoscope/transit-to-prompt.ts` with `transitAndNatalToPromptText(transitPlanets, natalCalculation, transitAspects)`:
    - Takes transit planet positions (from daily_transits cache or fresh calculation), the natal chart_calculations row data, and the computed transit aspects array.
    - Formats as structured text listing: (a) today's transit positions with sign and degree, (b) natal planet positions with sign, degree, and house, (c) active transit aspects with orb and applying/separating status.
    - Import and use `PLANETS_BG` and `ZODIAC_SIGNS_BG` from `@celestia/astrology` for Bulgarian planet/sign names in the prompt.
    - Import `ASPECTS_BG` for Bulgarian aspect names.

    3. Create `apps/web/app/api/horoscope/generate/route.ts` — POST endpoint:
    - Follow the EXACT pattern of `/api/oracle/generate/route.ts` (auth check, chart ownership, streaming, onFinish cache).
    - Flow:
      a. `auth()` check — 401 if not authenticated
      b. Parse body: `{ chartId }` (no topic needed — daily horoscope is a single reading)
      c. Resolve today's date in Sofia timezone: `new Intl.DateTimeFormat('en-CA', { timeZone: 'Europe/Sofia' }).format(new Date())` — this is the cache key
      d. Accept optional `?date=YYYY-MM-DD` query param to support "yesterday" lookups (validate format, only allow today or yesterday)
      e. Chart ownership check via supabase (same as oracle)
      f. Cache check: `daily_horoscopes` WHERE chart_id = chartId AND date = today → if found, return `{ content, cached: true, generatedAt }`
      g. Get/cache transit positions: check `daily_transits` WHERE date = today → if miss, `calculateDailyTransits(new Date())` and insert
      h. Load natal chart_calculations for the chartId
      i. Compute transit aspects via `calculateTransitAspects(transitData, natalPlanets)`
      j. Build prompt via `buildDailyHoroscopePrompt()` + `transitAndNatalToPromptText(...)`
      k. `streamText()` with `google('gemini-2.5-flash')`, temperature 0.85, maxOutputTokens 1500
      l. `onFinish`: insert into `daily_horoscopes` (chart_id, user_id, date, content, model_version)
      m. Return `result.toTextStreamResponse()`
    - Set `export const maxDuration = 60`
    - Use `createServiceSupabaseClient` (same import as oracle)
    - For the date param: if the requested date is yesterday and no cached horoscope exists for yesterday, return `{ content: null, unavailable: true }` (do NOT generate yesterday's horoscope retroactively — per research decision)
